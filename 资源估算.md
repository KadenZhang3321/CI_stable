# 监控体系资源估算

> 估算基准：**10 个业务集群**，每集群 **50 个节点**，共 500 节点。HA 模式，60s 抓取间隔，450K 活跃 series。
>
> 方案 A：告警优先，7 天保留。方案 B：告警 + 历史分析，30 天保留（CPU/内存与 A 相同，仅磁盘不同）。

---

## 一、中心集群

| 组件 | 实例 | CPU request | CPU limit | 内存 request | 内存 limit | 磁盘 (方案 A) | 磁盘 (方案 B) |
|------|------|-------------|-----------|-------------|-----------|--------------|--------------|
| Prometheus | 2 (HA) | 4C ×2 | 8C ×2 | 16Gi ×2 | 32Gi ×2 | 25Gi ×2 | 80Gi ×2 |
| Alertmanager | 2 (Gossip) | 0.1C ×2 | 0.5C ×2 | 128Mi ×2 | 256Mi ×2 | — | — |
| Grafana | 1 | 0.2C | 0.5C | 512Mi | 1Gi | 10Gi | 10Gi |
| Pushgateway | 1 | 0.1C | 0.2C | 128Mi | 256Mi | — | — |

| 中心集群合计 | CPU request | CPU limit | 内存 request | 内存 limit | 磁盘 |
|-------------|-------------|-----------|-------------|-----------|------|
| **方案 A** | **8.5C** | **17.2C** | **33.0Gi** | **65.8Gi** | **60Gi** |
| **方案 B** | **8.5C** | **17.2C** | **33.0Gi** | **65.8Gi** | **170Gi** |

> Prometheus 4C/16Gi 按 450K series 推导。2C/8Gi 是通用基线（小规模），4C/16Gi 给了 compaction 峰值余量。

## 二、每个业务集群

| 组件 | 部署方式 | CPU request | 内存 request | 说明 |
|------|---------|-------------|-------------|------|
| kube-state-metrics | Deployment ×1 | 150m | 200Mi | 50 节点集群 k8s 对象较多，适当上调 |
| node-exporter | DaemonSet (50 副本) | 10m ×50 | 64Mi ×50 | DaemonSet 无需预留 CPU，10m 为统计值 |
| Pushgateway | Deployment ×1 | 50m | 64Mi | 可选，如直推中心可省 |
| CronJob 巡检集 (×6) | 按需 | 0 | 0 | 执行时临时占用，结束释放 |

| | 单集群 | **10 集群合计** |
|---|--------|----------------|
| CPU request | **700m** | **7.0C** |
| 内存 request | **3.5Gi** | **35.0Gi** |

> node-exporter 是资源主体：单集群 500m CPU + 3.2Gi 内存。如果设 50m/节点（常见默认值），CPU 会到 2.5C/集群。这里用 10m 是实际水位。

## 四、方案 A vs B 差异

两个方案的 CPU 和内存完全一样（差异 < 10%，可忽略），**唯一区别是中心 Prometheus 磁盘**：

```
方案 A: 25Gi × 2 = 50Gi → 中心总计 60Gi
方案 B: 80Gi × 2 = 160Gi → 中心总计 170Gi
```

> 方案 B 多 110Gi 磁盘，换来 30 天历史趋势分析、月环比、容量预测能力。
