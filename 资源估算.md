# 监控体系资源估算

> 估算基准：**10 个业务集群**，每集群 **5 个节点**，共 50 节点。HA 模式，30s 抓取间隔。
>
> 活跃 series 拆分：
> - kube-state-metrics：~7K/集群 × 10 = **70K**
> - node-exporter：~600/节点 × 50 = **30K**
> - CronJob Pushgateway：可忽略
> - **合计 ≈ 100K**

---

## 一、方案概述

| | 方案 A：告警优先 | 方案 B：告警 + 历史分析 |
|---|---|---|
| 设计目标 | 故障发现和通知，不关心历史 | 告警 + 趋势分析 + 容量规划 |
| Prometheus 数据保留 | **7 天** | **30 天** |
| 长期存储 (Thanos/VM) | 无 | 可选（见第三节） |
| 适用阶段 | 快速上线、预算紧张 | 生产长期运行 |

---

## 二、Prometheus 资源推导

Prometheus 的三类资源消耗机制不同，分开算：

| 资源 | 受什么影响 | 7 天 vs 30 天差异 |
|------|-----------|-------------------|
| **内存** | 活跃 series 数量（head block，最近 2-3h） | **无影响** — 历史数据在磁盘压缩块里，不占内存 |
| **CPU** | 抓取解析 + 规则计算 + 定期压缩（compact） | **差异 < 10%** — 30 天多压几个旧块，可忽略 |
| **磁盘** | 活跃 series × 抓取间隔 × 保留天数 | **线性增长** — 唯一实质性差异 |

### 2.1 单实例 Prometheus 规格推导

```
日摄入量 = 100K series / 30s × 86400 × 1.5 bytes(压缩后) ≈ 432 MB
```

**内存推导：** head block 持有 ~2.5h 窗口内的全部样本在内存中做索引和压缩。100K series 在 Prometheus 中典型占用 8-12Gi（取决于标签基数）。request 14Gi，limit 24Gi 留 compaction 峰值余量。

**CPU 推导：** 100K series / 30s 抓取 ≈ 3,333 samples/s。按 1 core 处理 ~25K-35K series 的常规负载，取 3C request / 6C limit（limit 应对 compaction 尖峰）。

**磁盘推导：**

```
实际分配 = 日摄入 × 保留天数 × 1.6(索引+WAL) × 2(峰值余量)
```

| | 方案 A (7天) | 方案 B (30天) |
|---|---|---|
| 日摄入 | 432 MB | 432 MB |
| 原始累计 | 432 × 7 ≈ 3.0 GB | 432 × 30 ≈ 13.0 GB |
| × 开销系数 1.6 | 4.8 GB | 20.7 GB |
| × 峰值余量 2 | 9.7 GB → **15 Gi** | 41.5 GB → **50 Gi** |

| 单实例 Prometheus | CPU request/limit | 内存 request/limit | 磁盘 A | 磁盘 B |
|-------------------|-------------------|---------------------|--------|--------|
| | 3C / 6C | 14Gi / 24Gi | **15Gi** | **50Gi** |

## 三、全量资源汇总（HA 模式）

### 3.1 中心集群

| 组件 | 实例数 | CPU request/limit | 内存 request/limit | 磁盘 (方案 A) | 磁盘 (方案 B) |
|------|--------|-------------------|---------------------|--------------|--------------|
| Prometheus | 2 (HA) | 3C×2 / 6C×2 | 14Gi×2 / 24Gi×2 | 15Gi×2 | 50Gi×2 |
| Alertmanager | 2 (Gossip) | 0.1C×2 / 0.5C×2 | 128Mi×2 / 256Mi×2 | — | — |
| Grafana | 1 | 0.2C / 0.5C | 256Mi / 512Mi | 10Gi | 10Gi |
| Pushgateway | 1 | 0.05C / 0.2C | 64Mi / 128Mi | — | — |

| 中心集群合计 | CPU request | 内存 request | 磁盘 |
|-------------|-------------|-------------|------|
| **方案 A** | **6.45C** | **28.6Gi** | **40Gi** |
| **方案 B** | **6.45C** | **28.6Gi** | **110Gi** |

### 3.2 业务集群（方案 A/B 完全相同）

| 组件 | CPU request | 内存 request | 部署模式 |

| CronJob 巡检集 (×6) | burst | burst | 按需调度 |

|---|--------|----------------|
| CPU request | 0.4C | **4.0C** |
| 内存 request | 0.96Gi | **9.6Gi** |

### 3.3 总计

| | 中心集群 | 10 个业务集群 | **总计** |
|---|---------|-------------|---------|
| CPU request | 6.45C | 4.0C | **~11C** |
| CPU limit | 14.7C | — | **~19C** |
| 内存 request | 28.6Gi | 9.6Gi | **~39Gi** |
| 内存 limit | 51.6Gi | — | **~62Gi** |
| 磁盘 (方案 A) | 40Gi | — | **40Gi** |
| 磁盘 (方案 B) | 110Gi | — | **110Gi** |

> 方案 A/B 的 CPU 和内存完全一样，差异仅在磁盘。方案 B 多 70Gi。

## 四、Grafana 看板能力对比

| 看板功能 | 方案 A (7天) | 方案 B (30天) |
|---------|-------------|--------------|
| 实时指标 / 当前值 | 有 | 有 |
| 告警 firing 状态 | 有 | 有 |
| 7 天内趋势图 | 有 | 有 |
| 30 天趋势图 | **无**（图表右半空白） | **有** |
| 月环比 | **无** | **有** |
| 磁盘增长率推算 | 不准（窗口太短） | **可信** |
| 月度发布前后对比 | 仅最近一次 | 可对比最近 4 次 |
| 月度容量报告 | 无 | 可生成 |

典型影响：磁盘从 85% 涨到 95% 用了 20 天——方案 A 根本看不到这个趋势起点，方案 B 能拉出整条曲线，精确判断是哪个时间点开始加速增长的。

---

## 五、推荐

直接选方案 B。多 70Gi 磁盘成本极低，但数据窗口从 7 天拉到 30 天，运维能力完全不同。而且从 A 切 B 只能改 retention 重启，历史数据不会回溯。
