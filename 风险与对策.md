# 监控体系风险与对策

**关联文档：** [设计文档 v3.1](多集群基础设施主动监控体系设计.md) · [实施计划](实施计划.md) · [资源估算](资源估算.md)

---

## 一、Meta-monitoring：监控系统自己被监控

**风险：** 中心 Prometheus 或 Alertmanager 宕机，监控体系自身是盲的，没人知道告警已停摆。

**对策：**

- 在中心集群外部（另一台机器/另一朵云/办公网机器）部署独立拨测脚本，定时探测：
  - Prometheus `/-/healthy`
  - Alertmanager `/-/healthy`
  - Pushgateway `/metrics`
  - 企业邮箱 SMTP 连通性
- 该拨测的告警走**与邮件完全独立的通道**（直接调办公软件 IM webhook 或短信网关），确保邮件系统挂了也能通知到人
- 拨测间隔建议 1m，比业务告警更密

---

## 二、CronJob 巡检静默失败

**风险：** CronJob Pod 没调度、镜像拉不下来、脚本半路报错退出——都不会 push 新指标到 Pushgateway，旧值一直留着，告警永远不会触发。

**对策：**

- **心跳指标：** 每个 CronJob 脚本最后一行无论成功失败都 push `cronjob_last_run_timestamp`，带上 cluster 标签
- **心跳超时告警：** 中心 Prometheus 配规则，`time() - cronjob_last_run_timestamp > 期望周期 × 2` → WARNING
- **Job 失败告警：** 加一条规则 watch `kube_job_status_failed`，失败数 > 0 即通知
- 这两条规则与 4.5 节的 Pushgateway 新鲜度校验互补，覆盖"根本不推"和"推了但旧"两个场景

---

## 三、SMTP 单点故障

**风险：** 邮箱密码过期、SMTP 服务器限流或宕机——Alertmanager 能生成告警但发不出去，完全无感知。

**对策：**

- Alertmanager 内置指标 `alertmanager_notifications_failed_total` 配告警规则，发送失败增量 > 0 立即触发
- Alertmanager 配置中加一个 webhook receiver 作为备用通知通道，指向办公软件机器人的 webhook URL
- SMTP 密码接近过期时主动续期或提前通知管理员
- Meta-monitoring（第一条）的 SMTP 连通性拨测兜底

---

## 四、机器人解析脆弱

**风险：** 改动了 `alertname`、调整了邮件模板字段顺序——机器人正则解析即告失效，运维人员突然收不到通知。

**对策：**

- 监控侧与办公软件团队约定标题格式版本号（如 `v1`），任何格式变更必须双方评审并同步发布
- 部署一条 Canary 告警：定时触发、固定内容，机器人侧每日检查是否正确解析并分发，解析失败报警给机器人团队而不是运维团队
- Alertmanager 邮件模板和标题格式作为代码管理，归入 Git，变更走 MR 评审

## 五、告警风暴与多集群抑制

**风险：** 9 条规则 × 10 集群 = 最多同时 firing 90 条。如果 GitHub 真的全球宕了，10 个集群同时报 GithubUnreachable + RunnerDown + MirrorSyncFailed，邮件刷屏。

**对策：**

- **Alertmanager inhibit（告警抑制）：** 根因告警抑制衍生告警
  ```yaml
  # GitHub 不可达时，抑制 RunnerDown 和镜像同步失败
  inhibit_rules:
    - source_match:
        alertname: GithubUnreachable
      target_match_re:
        alertname: "RunnerDown|MirrorSyncFailed"
  ```
- **分组合并：** `group_by: ['alertname']`，同类型告警 10 个集群的实例合并到一封邮件正文中列出
- **频率控制：** `group_interval: 5m`，`repeat_interval: 4h`，同一告警至少间隔 4 小时才重发

## 六、Prometheus HA 配置漂移

**风险：** HA 对的两个 Prometheus 实例独立运行，手动改其中一个的规则/抓取目标忘了改另一个，导致告警行为不一致。

**对策：**

- 两个实例挂同一个 ConfigMap，规则文件、抓取目标统一存储
- 引入 GitOps（ArgoCD / Flux），所有配置在 Git 中管理，提交即自动推送到两个实例
- 加一条 Prometheus 自检规则：`prometheus_tsdb_head_series` 两个实例差异 > 20% → 告警

---

## 七、CronJob 巡检脚本批量升级管理

**风险：** 10 个集群各 6 个 CronJob = 60 个资源，脚本升级时逐个集群 `kubectl apply` 容易遗漏。

**对策：**

- 全部 CronJob 用同一套 Helm Chart 或 Kustomize overlay 管理
- 脚本作为 ConfigMap 模板，按 cluster 标签渲染
- 推送到 Git，ArgoCD ApplicationSet 自动同步到所有业务集群的 monitoring 命名空间
- 每次变更用 `kubectl get cronjob -n monitoring --context <cluster>` 批量验证

---

## 八、TLS 证书轮换

**风险：** 中心到 10 个集群的 TLS/mTLS 链路涉及至少 10 张 exporter 端证书。证书过期后抓取全断，告警静默。

**对策：**

- 各集群部署 cert-manager，exporter 端证书由 cert-manager 自动签发和续期
- 中心 Prometheus 抓取配置中 `ca_file` 指向 CA 证书，服务端证书轮换后自动信任
- 加一条证书到期提前告警：
  ```promql
  # 通过 blackbox-exporter 探测 TLS 证书到期天数
  probe_ssl_earliest_cert_expiry - time() < 86400 * 30
  ```

---

## 九、Grafana 未认证

**风险：** 目前设计中 Grafana 未提及认证。能访问中心集群的人就能看所有集群的指标面板。

**对策：**

- 关闭匿名访问：`GF_AUTH_ANONYMOUS_ENABLED=false`
- 接入 OAuth2/OIDC（GitHub OAuth、社区 LDAP 或 Keycloak），复用现有身份体系
- 最小化默认角色：新用户默认 Viewer，禁止随意修改仪表盘
- 管理员凭据存入 Kubernetes Secret，不硬编码

## 十、优先级排序

| 优先级 | 风险 | 理由 |
|--------|------|------|
| **P0 必须做** | Meta-monitoring | 监控死了没人知道 = 白建 |
| **P0 必须做** | SMTP 单点 + 告警风暴抑制 | 告警太多或收不到 = 白告 |
| **P1 尽快做** | CronJob 静默失败 + Grafana 认证 | 第一个会导致漏报，第二个是安全基线 |
| **P2 可以稍后** | 配置漂移、批量升级、TLS 轮换 | 提效项，手工也能撑一段时间 |
| **P3 对齐即可** | 机器人解析脆弱 | 对接阶段跟办公软件团队约定好规范就行 |
