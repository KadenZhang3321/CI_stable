| # | 任务 | 产出 | 验证方式 |

## Phase 1：中心集群搭建

**目标：** 在中心集群部署 Prometheus（HA 对）、Alertmanager（Gossip 集群 ×2）、Grafana，全部可通过 `kubectl port-forward` 或 Ingress 访问。

### 1.1 部署 Alertmanager 集群

> **为什么先部署 Alertmanager：** Prometheus 启动时如果引用了不存在的 Alertmanager，不会阻断但会持续报错。先搭 Alertmanager 避免这个问题。Gossip 集群的 `cluster.peer` 需要 Pod 启动前就知道至少一个对端的地址，用 StatefulSet + headless Service 最省事。

| # | 任务 | 说明 |
|---|------|------|
| 1.1.1 | 编写 `alertmanager-config.yaml` | 全局配置：SMTP 服务器、收件人、邮件模板路径。注意 `group_by: ['cluster', 'alertname']`，`group_interval: 5m` |
| 1.1.2 | 编写 `alertmanager-statefulset.yaml` | StatefulSet，replicas=2。启动参数 `--cluster.peer=alertmanager-0.alertmanager:9094`，端口 9093(API)/9094(gossip mesh) |
| 1.1.3 | 编写 `alertmanager-svc.yaml` | Headless Service + ClusterIP Service 双 Service 模式 |
| 1.1.4 | 部署并验证 Gossip 集群 | `kubectl logs alertmanager-0` 应看到 `peer alertmanager-1 joined`。访问 `/api/v2/status` 确认 2 节点 |

### 1.2 部署 Prometheus HA 对

| # | 任务 | 说明 |
|---|------|------|
| 1.2.1 | 编写 `prometheus-config.yaml` | 基础配置。scrape 目标暂时为空（Phase 2 才添加业务集群 job）。`evaluation_interval: 30s`。规则文件路径预留给 Phase 4 |
| 1.2.2 | 编写 `prometheus-statefulset.yaml` | StatefulSet，replicas=2。两个实例独立抓取（不共享 TSDB），`--storage.tsdb.retention.time=15d`，预留 PVC 50Gi |
| 1.2.3 | 编写 `prometheus-svc.yaml` | Headless + ClusterIP |
| 1.2.4 | 部署并验证 | 两个 Pod Running，`/targets` 页面无报错 |

### 1.3 部署 Grafana

| # | 任务 | 说明 |
|---|------|------|
| 1.3.1 | 编写 `grafana-deployment.yaml` | 单实例 Deployment，PVC 持久化仪表盘。环境变量配置 Prometheus 数据源地址（指向 Prometheus Service） |
| 1.3.2 | 部署并导入基础仪表盘 | 导入 Node Exporter Full、Kubernetes Cluster 等社区仪表盘，验证数据源连通 |

### 1.4 部署中心 Pushgateway

| # | 任务 | 说明 |
|---|------|------|
| 1.4.1 | 编写 `pushgateway-deployment.yaml` | 单 Deployment，关键巡检 CronJob 将直推到此实例 |
| 1.4.2 | 部署 | `kubectl apply -f` |

## Phase 2：首个业务集群接入

**目标：** 选一个非生产集群（如 `staging`）作为试点，完整跑通采集 → 中心抓取 → 告警 → 邮件全链路。

### 2.1 集群侧部署采集器

> 所有 YAML 统一放到 `k8s/monitoring/` 目录，以 cluster 标签命名区分。

| # | 任务 | 说明 |
|---|------|------|
| 2.1.1 | 编写 `kube-state-metrics-deployment.yaml` | 注意 `--metric-labels-allowlist` 只放行需要的 label。通过 `external_labels: {cluster: staging}` 注入集群标识 |
| 2.1.2 | 编写 `node-exporter-daemonset.yaml` | DaemonSet 每个节点一份。External label 同样注入 `cluster` |
| 2.1.3 | 编写 `pushgateway-deployment.yaml` | 集群侧 Pushgateway（可选，见设计 4.4 节） |
| 2.1.4 | 全部部署到 `monitoring` 命名空间并验证 | `kubectl get all -n monitoring` 确认 Running |

### 2.2 配置中心 Prometheus 抓取

| # | 任务 | 说明 |
|---|------|------|
| 2.2.1 | 编写 `targets-staging.yaml`（`file_sd_configs`） | 包含 staging 集群的 kube-state-metrics、node-exporter、pushgateway 端点 URL。静态附加 `cluster: staging` 标签 |
| 2.2.2 | 中心 Prometheus 加载并验证 | 热加载 `curl -XPOST .../-/reload`，`/targets` 显示 staging 的 3 个 job 均为 UP |

## Phase 3：巡检脚本开发

**目标：** 为 9 个巡检项逐一编写 CronJob 脚本，部署到 staging 集群试运行。

> 脚本用 Bash + curl 实现，结果以 Prometheus textfile 格式输出，通过 `curl -X PUT` 推送到 Pushgateway。每个脚本是一个 ConfigMap，CronJob 挂载后执行。

### 3.1 GitHub 不可达（编号 1）

| # | 任务 | 说明 |
|---|------|------|
| 3.1.1 | 编写 `probe-github.sh` | `curl -o /dev/null -s -w '%{http_code}' https://github.com`，0/1 转换为 `github_probe_success{cluster="X"} 1\|0` |
| 3.1.2 | 编写 CronJob `cronjob-github-probe.yaml` | 每 1m 执行，推送到 Pushgateway |

### 3.2 ServiceAccount 权限过大（编号 2）

| # | 任务 | 说明 |
|---|------|------|
| 3.2.1 | 编写 `audit-sa.sh` | 遍历所有 NS 的 ServiceAccount，检查是否绑定了 cluster-admin 或过宽的 ClusterRole。输出 `sa_privilege_excess{cluster="X", namespace="Y", sa="Z"} 1` |
| 3.2.2 | 编写 CronJob `cronjob-sa-audit.yaml` | 每日执行一次 |

### 3.3 云账号欠费/冻结（编号 3）

| # | 任务 | 说明 |
|---|------|------|
| 3.3.1 | 编写 `check-cloud-account.sh` | 调用云厂商 BSS API（需配置 AK/SK），查询账户状态。输出 `cloud_account_status{cluster="X"} 1\|0` |
| 3.3.2 | 编写 CronJob | 每 10m 执行 |
| 3.3.3 | 凭据管理 | AK/SK 存入 Secret，CronJob 通过 env 引用 |

### 3.4 共享盘使用率过高（编号 4）

| # | 任务 | 说明 |
|---|------|------|
| 3.4.1 | 编写 `check-disk-usage.sh` | `df -P /shared` 取百分比，输出 `shared_disk_usage_percent{cluster="X", instance="Y"} 86` |
| 3.4.2 | 编写 CronJob | 每 5m 执行。需挂载 hostPath 或 PVC |

### 3.5 代理证书过期/代理宕机（编号 5）

| # | 任务 | 说明 |
|---|------|------|
| 3.5.1 | 编写 `check-proxy-cert.sh` | `openssl s_client -connect proxy:443` 提取证书到期日期，计算剩余天数。输出 `proxy_cert_days_left{cluster="X"}` |
| 3.5.2 | 编写 `check-proxy-reachable.sh` | 简单的 TCP 连通性检查。输出 `proxy_reachable` |
| 3.5.3 | 分别编写 CronJob | 证书日检，连通性 10m |

### 3.6 代码仓镜像同步失败（编号 6）

| # | 任务 | 说明 |
|---|------|------|
| 3.6.1 | 编写 `check-mirror-sync.sh` | 依赖具体镜像同步工具的输出（如 `gitea mirror`、`gitlab mirror`），解析最新同步状态 |
| 3.6.2 | 编写 CronJob | 日检或同步后触发 |

### 3.7-3.9：Prometheus 原生采集项

> OOM（7）、Runner 副本（8）、调度积压（9）不需要 CronJob — 数据来自 kube-state-metrics，直接在 Phase 4 写 PromQL 规则即可。

## Phase 4：告警规则编写

**目标：** 在中心 Prometheus 的 rule file 中编写 9 条告警规则，每条规则均通过 `cluster` 标签区分来源。

### 4.1 创建规则文件

| # | 任务 | 说明 |
|---|------|------|
| 4.1.1 | 创建 `rules/infrastructure.yaml` | 所有告警规则的 YAML 文件，挂载到 Prometheus |
| 4.1.2 | 编写 GitHub 不可达规则 | `github_probe_success == 0`，`for: 2m`，severity: CRITICAL |
| 4.1.3 | 编写 SA 权限过大规则 | `sa_privilege_excess > 0`，`for: 1h`，severity: WARNING |
| 4.1.4 | 编写云账号欠费规则 | `cloud_account_status == 0`，`for: 5m`，severity: CRITICAL |
| 4.1.5 | 编写共享盘使用率规则 | 两条规则：`> 85 for 5m` (WARNING) + `> 95 for 1m` (CRITICAL) |
| 4.1.6 | 编写代理证书/宕机规则 | 证书 `< 30` 天 (WARNING)，`proxy_reachable == 0 for 2m` (CRITICAL) |
| 4.1.7 | 编写镜像同步失败规则 | `mirror_sync_success == 0`，`for: 1h`，severity: WARNING |
| 4.1.8 | 编写 OOM 增量规则 | `increase(kube_pod_container_status_terminated_reason{reason="OOMKilled"}[10m]) > 0`，severity: INFO |
| 4.1.9 | 编写 Runner 副本规则 | `kube_deployment_status_replicas_available{deployment=~".*runner.*"} == 0`，`for: 2m`，severity: CRITICAL |
| 4.1.10 | 编写调度积压规则 | `kube_pod_status_phase{phase="Pending"} > 5`，`for: 5m`，severity: WARNING |

### 4.2 规则验证

| # | 任务 | 说明 |
|---|------|------|
| 4.2.1 | 语法检查 | `promtool check rules rules/infrastructure.yaml` |
| 4.2.2 | 在 staging 集群触发测试 | 人为制造条件（停 Runner、填磁盘、断 GitHub）确认告警进入 Alertmanager |

### 4.3 Alertmanager 路由与邮件模板

| # | 任务 | 说明 |
|---|------|------|
| 4.3.1 | 配置 Alertmanager `route` | 默认路由到 `infra-alert`。`repeat_interval: 4h` |
| 4.3.2 | 编写邮件模板 | 标题格式 `[{{ .Labels.severity }}][{{ .Labels.cluster }}] {{ .Labels.alertname }}{{ if .Labels.instance }}-instance-{{ .Labels.instance }}{{ end }}` |
| 4.3.3 | 端到端测试 | 触发一条告警 → 确认收到邮件 → 确认标题格式正确 |

## 依赖关系与并行度

```
Phase 0 ──► Phase 1 ──► Phase 2 ──► Phase 4 ──► Phase 5 ──► Phase 7
                │                        │
                └──► Phase 3 ────────────┘
                                             Phase 6 (可并行，依赖 5)
```

- Phase 1 和 Phase 3 可并行推进（中心集群搭建和巡检脚本开发分开做）
- Phase 6 依赖 Phase 5 的邮件链路跑通，但可以提前跟办公软件团队对齐接口
- Phase 4（告警规则）和 Phase 3（巡检脚本）有隐式依赖：规则引用的指标名必须在脚本中一致，建议先约定好指标名清单再并行开发
